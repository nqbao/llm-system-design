**System Design – Real‑time Chat (WhatsApp / Slack style)**  
> *Prepared for a product that needs to support millions of users with sub‑300 ms real‑time semantics, multiple chat channels, attachments, and long‑term history & search.*

---

## 1.  High‑level Architecture

```mermaid
graph TD
  %% Clients
  mobile((Mobile Client))
  web((Web Client))

  %% API side
  apigw[(API Gateway – HTTPS / WSS)]
  auth[(Auth Service – JWT)][style=fill:#f9f]
  chat[(Chat Service – handle writes)]
  pres[(Presence Service – user online status)]
  notif[(Notification|FCM/APNs)]

  %% Back‑end storage
  usrs[(Users – RDBMS)]
  chs[(Channels – RDBMS)]
  msgs[(Messages – NoSQL + Kafka)] 
  atch[(Attachments – S3)]
  estr[(ElasticSearch – search)]
  dk[(Datacenter‑level KV – Redis)]
  kaf[(Kafka Broker – topic “msg‑topic”)]

  %% Delivery
  wsnode[WebSocket Servers – horizontal cluster]
  msgsub[(MessageConsumer – Kafka -> WS)]
  
  %% Connections
  mobile --> apigw
  web --> apigw

  apigw --> auth
  apigw --> chat
  apigw --> pres
  apigw --> noti[Notification API]

  auth --> usrs
  chat --> usrs
  chat --> chs
  chat --> msgs
  chat --> atch
  funk[(Attachment Storage – S3)] --> atch
  msgs --> kaf
  kaf --> msgsub
  msgsub --> wsnode
  wsnode --> mobile
  wsnode --> web

  pres --> dk
  dk --> wsnode
  estr --> ?      %% Search service talks to msgs / estr
  mem[(Metrics & Logs – Prometheus / Loki)]
```

*Key take‑aways*  
* All user traffic enters via a single horizontally‑scalable **API Gateway** (REST + WebSocket).  
* **JWT** gives stateless auth; the gateway verifies it and injects the `user‑id`.  
* The **Chat Service** is the *write* entry point for new messages and attachments.  
* Messages are persisted **immediately** to a NoSQL store and published to a **Kafka** topic for real‑time delivery.  
* A background **Message Consumer** subscribes to Kafka, looks up online recipients using the **Presence Service** (Redis), and pushes via **WebSocket Servers**.  
* **Search** is powered by **ElasticSearch** which indexes the message contents (eventual consistency).  
* **Attachments** are stored in **S3**; only a small metadata record lives in the NoSQL store.

## 2.  Data Models

| Table / Collection | Primary keys | Replication | Notes |
|---------------------|--------------|-------------|-------|
| **Users** (RDBMS)  | `user_id` (UUID) | Master‑slave | Store email, phone, profile, last‑login |
| **Channels** (RDBMS) | `channel_id` (UUID) | Master‑slave | `type` = DM/Group, membership via join table |
| **Attachments** (RDBMS) | `attach_id` | Master‑slave | Includes `s3_uri`, `size`, `type` |
| **Messages** (NoSQL – Cassandra / DynamoDB) | Partition key = `channel_id`, clustering key = `seq` | 3‑node replication | Store `{msg_id, sender, timestamp, content, attachment_ids}` |
| **Presence** (Redis) | `user_id` | In‑memory only (replicated 3‑node) | `online_status`, `socket_id` |
| **Kafka topic** `chat‑messages` | `key = channel_id` | 3‑node cluster | Partitioned 200 ways, `seq` is offset |

*Message ID* is generated by the Chat Service as `(channel_id · 2^63) + local_seq`.  
`local_seq` is an increasing integer (Cassandra/LWT or atomic counter).  
Guarantees **ordering per channel** (up‑to). Since all consumers on a partition read in the same order, all members get the same sequence.

## 3.  Component‑to‑Component Work Flow (Sending a Message)

1. **Client → API Gateway**  
   *WebSocket upgrade receives JWT in header.*  
   Node validates JWT (using Auth Service); if valid, associates `user_id` with the connection.

2. **Client → Chat Service** (via WebSocket message)  
   Payload: `{“channel_id”: X, “text”: “hello”, “attachments”: […]}`
   * Insert into **NoSQL** (`channel_id`, `seq`, …).  
   * Persist attachment metadata and upload files to **S3** (async).  
   * Produce to **Kafka** topic `chat‑messages` (key=channel_id).  
   * Acks back to client (send‑ack with `msg_id`).

3. **Kafka → Message Consumer**  
   * Each consumer subscribes to one or more partitions (the cluster assigns them).  
   * For each record, fetch list of online members from **Presence(n)**.  
   * Construct WebSocket payload `{channel_id, msg_id, …}`.  
   * Push to all **WebSocket Servers** holding those users.

4. **WebSocket Servers → Clients**  
   * Maintain in‑memory map `socket_id → connection`.  
   * Use `socket_id` from Presence to find correct server (a sortable key).  
   * Send payload; ack on client ≤ 150 ms.

5. **Offline Users**  
   * If presence says user offline, consumer writes “pending” flag to the **NoSQL** row.  
   * Client fetches pending on reconnect.

## 4.  Capacity Math (Baseline + Peak)

| Metric | Baseline | Peak (×10) | Calculation |
|--------|----------|------------|-------------|
| **Registered users** | 10 M | – | – |
| **DAU** | 1 M | – | – |
| **Avg messages per user per day** | 50 | – | ⇒ 50 M messages/day |
| **Avg msg size** | 0.5 KB | – | 25 GB/day |
| **Messages per second** | 580 | 5800 | 50 M ÷ 86400 ≈ 580 |
| **Msgs/sec persistence** | 580 | 5800 | – |
| **Kafka write throughput** | 580 msg/s | 5800 msg/s | 200 partitions → 5.8 msg/s/partition (baseline); 29 msg/s/partition (peak) |
| **Persisted objects** | 50 M | – | – |
| **Attachments per message** | 10 % | – | 5 M attachments/day |
| **Avg attachment size** | 5 MB | – | 25 GB/day |
| **Upload throughput** | 5 GB/s | 5 GB/s | 5 MB × 1 msg/s |
| **WebSocket connections** | avg 200 k | peak 500 k | Each WS node holds ~10 k sockets → 20 nodes baseline, 50 nodes peak |
| **Message pushes per second** | 200 k | 1.2 M | Avg recipients 10 → 580 msg/s × 10 = 5.8 k pushes/s baseline, 58 k pushes/s peak |
| **Push bandwidth** | 1 MB/s | 10 MB/s | 200 bytes payload per push |

### Kafka
* 200 partitions, 3‑node cluster (each node holds ~66 partitions).  
* Replication factor = 3 → 600 “v‑nodes” overhead.  
* Broker CPU/~2 cores, 16 GB RAM each safe for 5‑10 k writes per second.

### NoSQL (Cassandra / DynamoDB)
* 3‑node cluster, 5 GB per node, 1 GB data per 100 k messages.  
* Each write stays <1 ms on one node (DRY).

### WebSocket nodes
* 32 GB RAM, 8 CPU cores.  
* Each node pushes ~1 k received messages/s => 10 k WS pushes per node ➝ 20 node baseline.

### Presence (Redis)
* 4‑node cluster, 16 GB RAM each.  
* Memory cost: 200 k online users → 200 k * 50 B ≈ 10 MB → well below footprint.

### S3
* 5 GB/s ingress per bucket (~3–4 Gbps). Multi‑AZ replication optional.

## 5.  Trade‑offs & Design Rationale

| Feature | Trade‑off | Rationale |
|---------|-----------|-----------|
| **Ordering** | Per‑channel ordering enforced by partition key & Kafka offset. | Guarantees do‑not‑mix, simple to implement. |
| **Latencies** | Deliver to online user in <200 ms, offline history optional. | Real‑time requirement; offline retrieval handled by fetch API. |
| **CAP** | *AP* – Availability & Partition tolerance favored over strict consistency. | Quick writes and reads under network partitions; consistency at the level of message stores is strong enough for chat. |
| **Storage** | Attachments in cheap S3, messages in NoSQL. | Minimize cost per GB while keeping write speed. |
| **Scalability** | Horizontal scaling of every layer. | Each component can be scaled independently to handle spikes. |
| **Complexity** | Kafka + multiple stores increases ops surface. | Allows decoupled consumption and real‑time streaming, giving robust failure isolation. |
| **Search** | ElasticSearch eventually consistent; query latency can be a few hundred ms. | Users rarely need instant search; acceptable. |

## 6.  Failure Modes & Mitigations

| Component | Failure | Impact | Mitigation |
|-----------|---------|--------|------------|
| **API Gateway** | Node crash | Small outage, queued connections last until replacement processes | Keep stateless; rapid logstash health checks; use a health‑check ready LB |
| **WebSocket Server** | Crash mid‑connection | Latency spike, missed messages | Use sticky session via `user_id`; upon reconnect fetch pending messages |
| **Auth Service** | Down | Zero new connections | JWT still valid; existing connections unaffected|
| **Presence (Redis)** | Node failure | Loss of online status | 3‑node cluster; if one down, still 2 nodes reachable; fallback to DB |
| **Kafka Broker** | Leader crash | Message loss or stall | Replication factor 3; leader election; produce with `acks=all` |
| **NoSQL (Cassandra)** | Node down | Write throughput drop | Replicated 3‑node; read/write to three replicas, one always available |
| **S3** | Storage tier unavailable | Attachment download blocked | Multi‑AZ bucket; S3 continuity is strong |
| **ElasticSearch** | Shard lost | Search failures | Replica shards = 1; data duplicated; re-creation on failover |
| **Network Partition** | Application part blocked | Users offline | Circuit breakers; graceful degradation (show “offline” & queue up) |
| **CPU spike** in consumer | Lag in pushes | Message delay | Back‑pressure, pause consumer, fill slack queue; autoscale consumer fleet |

**Chaos‑testing strategy**  
- Randomly kill a WebSocket node → ensure reconnect, pending message retrieval.  
- Simulate Kafka leader failover → note message replay & offset lag.  
- Stop Redis node → presence fallback to DB and eventual consistency.  
- Cut off S3 region → confirm fallback to alternative region.

## 8.  Tech Stack (Sample)

| Layer | Technology | Why |
|-------|------------|-----|
| • **Client** | Web: React + Socket.io; Android / iOS: native | Cross‑platform WebSocket support |
| • **API Gateway** | Kong / NGINX + Lua | SSL/TLS termination, rate limiting, JWT validation |
| • **Auth** | Go / SpringBoot | JWT + OAuth2 provider (Okta/GitHub) |
| • **Chat Service** | Go (gRPC) | Low latency, high concurrency |
| • **Presence** | Redis Cluster | In‑memory, fast, replicated |
| • **Message Queue** | Kafka (Confluent) | Exactly‑once semantics, order guaranteed |
| • **Message Store** | Cassandra (or DynamoDB) | Partitioned, low‑latency writes |
| • **File Store** | AWS S3 (or MinIO) | Durable, cheap, global |
| • **Search** | ElasticSearch | Full‑text indexing, scalable |
| • **WebSocket** | Node/Go + `ws` or `gorilla/websocket` | Light‑weight, horizontally scalable |
| • **Monitoring** | Prometheus + Grafana + Loki | Open‑source, widely adopted |

---

### Final Note

This design satisfies the core requirements of a **real‑time, scalable, fault‑tolerant chat**:

1. **Fast, ordered delivery** for online users → Kafka + WebSocket consumer.  
2. **Durable history** + searchable → NoSQL + ElasticSearch.  
3. **Attachment scalability** → S3 + metadata in the store.  
4. **Low‑cost growth** → open‑source tech, cheap on‑demand services.  
5. **Robustness** → replication, failure patterns, retriable writes, graceful degradation, and chaos‑testing‑driven resilience.
