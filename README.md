# LLM System Design Benchmark

A benchmark for evaluating LLM system design capability — quickly compare how well different models produce solid, production-ready system designs.

See the live site at [nqbao.github.io/llm-system-design](https://nqbao.github.io/llm-system-design/).

## Setup

- **9 questions** across 4 tiers: easy (URL shortener), medium (chat, news feed, Twitter), hard (rate limiter, job scheduler, YouTube), chaos (metrics pipeline, object store)
- **9 models**: claude-sonnet-4.6, deepseek-v4-pro, gemini-3.1-pro, gemma-4-31b-it, gpt-5.4, gpt-oss-120b, gpt-oss-20b, kimi-k2.6, minimax-m2.7
- **3 judges**: claude-sonnet-4.6, deepseek-v4-pro, kimi-k2.6 — each scores every transcript independently

Each model receives a **cold prompt** (generic system design instructions) followed by a **question prompt**. All models run with thinking enabled at **reasoning effort = high**.

### How it runs

1. **Interview** — send a cold design prompt to each model. Output lands in `runs/<model>/`.
   ```bash
   python3 run_interview.py --model <model> --question all
   ```

2. **Judge** — a separate LLM scores each transcript on 5 dimensions. Saves to `judgments/<judge>/`.
   ```bash
   python3 run_judges.py --model <judge> --transcript runs/<model>/<question>_cold.json
   ```

3. **Report** — reads all transcripts and judgments from disk, prints leaderboard.
   ```bash
   python3 eval.py
   ```

### Build the site

The static site is built with Astro Starlight. All markdown content is generated by `build_site.py`.

```bash
rm -rf leaderboard/starlight/src/content/docs
python3 leaderboard/build_site.py
cd leaderboard/starlight && npm run build
```

Output goes to `leaderboard/dist/`.

## Scoring Dimensions

| Dimension | What it measures |
|---|---|
| `requirements_scoping` | Did it start with a scoped problem statement? Quantified SLAs? Non-goals? |
| `capacity_estimation` | Did the math happen AND drive design choices? |
| `architecture_coherence` | Do components serve real requirements? Hallucinations? |
| `deep_dive_depth` | Did it go deep on subsystems with concrete mechanisms and tradeoffs? |
| `tradeoffs_failure_modes` | Did it acknowledge choices and think about what breaks? |

Scores: 0–5. 0 = missing/refused/truncated. 3 = adequate. 5 = exceptional (rare).

## Caveats

- Judges are not calibrated against human feedback
- All models run through a LiteLLM gateway — results may differ from direct API access
- Each model receives only the cold prompt; no multi-turn, no follow-up
- Judge models are also in the benchmark (a judge scoring its own transcript may bias results)
